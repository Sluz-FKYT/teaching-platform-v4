# 学生端实验答题工作台完全改造设计

## 目标

将学生端实验答题页从“整页滚动 + 通用文本输入”重构为**固定视口的双区答题工作台**，满足以下目标：

1. 页面不再以浏览器整页滚动作为主要交互方式。
2. 左侧题目导航区与右侧答题区各自独立滚动。
3. 不同题型使用不同的答题样式，而不是统一退化为文本框。
4. 填空题在展示题干文本的同时，将 `【填空1】`、`【填空2】` 等占位替换为可直接输入的内联下划线输入框。
5. 学生端保存链路正式接入结构化答案 JSON，同时兼容当前依赖 `answerText` 的评分与展示逻辑。
6. 改造后页面仍符合真实前后端、真实数据库、真实权限、真实测试约束，不引入 mock 闭环。

## 背景与现状

当前学生端实验答题页入口为 `frontend/src/views/student/LabSteps.vue`，路由为 `/student/labs/:id/steps`。

基于本次代码审查，现状存在以下核心问题：

1. 页面根节点没有接入锁定工作台的高度壳，导致滚动落在浏览器整页上，而不是题目导航区和答题区内部分治。
2. 学生端详情接口返回的题项结构只有 `questionType / content / answerText` 等基础字段，没有返回客观题选项、填空 blanks、代码语言等结构化元数据。
3. 学生端前端状态模型是 `Record<number, string>`，保存时只提交 `answerText`，所以单选、多选、填空都被迫退化成“文本协议”。
4. 后端保存接口虽然预留了 `answerPayloadJson`，数据库也有 `answer_json` 字段，但现行主流程并未真正写入或消费这些结构化答案字段。
5. 教师端实验步骤编辑器与题库题型编辑器已经拥有结构化题型规则，例如 choice `options`、fill `blanks` 与题干 token，但学生端并未消费这套结构。

## 设计结论

本次采用以下总体方案：

- 将学生端 `LabSteps.vue` 重构为**固定视口的双区答题工作台**。
- 学生端实验详情接口返回题型驱动的结构化答题元数据。
- 学生端按题型维护结构化答案状态，不再使用统一字符串草稿模型。
- 保存接口同时接收并持久化 `answerPayloadJson` 与兼容文本 `answerText`。
- 第一阶段评分与回显继续兼容 `answerText`，但新数据从本次起正式进入 `answer_json` 轨道。

该方案对应本次确认的“完全改造”与推荐的“方案 A”。

## 一、页面结构设计

### 1. 页面整体布局

页面改为两层结构：

1. **压缩头部（task header）**
2. **固定高度的 answer workbench 主体**

头部保留：

- 返回实验详情
- 实验标题 / 当前题号 / 题目总数 / 已保存进度
- 当前题保存状态
- 学生身份与退出入口

移除当前会显著吞噬可视高度的松散纵向堆叠节奏，让用户进入页面后立刻同时看到左侧导航和右侧答题主体。

### 2. 双区分栏

#### 左栏：题目导航区

职责：切换题目、展示进度、快速识别每题状态。

包含内容：

- 题目总数与实验小结状态
- 已保存进度条
- 题项列表
- 每题的序号、题型、分值、作答状态

要求：

- 左栏固定宽度。
- 顶部进度摘要固定，题项列表 body 独立滚动。
- 切题时左栏不离开视野。
- 当前题项高亮，已保存、未作答、已完成状态语义清晰。

#### 右栏：答题工作区

职责：展示当前题题干、渲染对应题型交互、承载实验小结与底部操作栏。

包含内容：

- 当前题头部信息（题号、题型、分值、状态）
- 题目正文
- 按题型渲染的作答组件
- 实验小结区域
- 底部操作栏（上一题 / 下一题 / 保存当前题 / 提交实验）

要求：

- 右栏为主区域，占据页面主要宽度。
- 右栏内部采用“头部信息 + 可滚动作答主体 + sticky 底栏”。
- 切换题目时右栏骨架不变，仅题目内容和题型组件切换。

## 二、滚动与视口规则

### 1. 滚动归属

本次改造的第一优先级规则如下：

- 浏览器整页不再是主要滚动容器。
- 学生实验答题页主体必须锁定在视口可用高度内。
- 滚动仅允许出现在：
  - 左栏题目列表 body
  - 右栏答题主体 body

### 2. 具体实现原则

1. 页面根节点接入与工作台页面一致的锁定高度壳层，而不是继续使用普通长页面流布局。
2. `LabSteps.vue` 的主工作台区域必须使用 `min-height: 0` 与 `overflow: hidden` 阻断整页滚动继续传导。
3. 左右两栏都采用 `display: flex; flex-direction: column;`，由各自 body 区承担 `overflow-y: auto`。
4. 底部操作栏固定在右栏底部，不能随着右栏内容滚动脱离视野。

### 3. 用户可感知结果

无论当前题是长题干代码题，还是多空位填空题，用户都应该始终看到：

- 左侧题目导航与进度
- 右侧当前题干与答题区
- 底部操作栏

页面不会再因为某道题内容变长而变成整页长卷轴。

## 三、分题型答题交互设计

### 1. 单选题

单选题使用单选组交互，而不是输入字母。

设计要求：

- 渲染完整选项列表。
- 每个选项支持点击整行选中。
- 当前选中态有明确视觉反馈。
- 页面保存时同步维护结构化答案与兼容文本答案。

结构化答案形态：

```json
{ "selectedKey": "A" }
```

兼容文本形态：

```text
A
```

### 2. 多选题

多选题使用复选组交互，而不是输入 `A,C` 之类的文本。

设计要求：

- 渲染完整选项列表。
- 支持点击整行切换勾选状态。
- 保持稳定顺序，避免保存后回显顺序抖动。
- 保存前生成兼容 `answerText`。

结构化答案形态：

```json
{ "selectedKeys": ["A", "C"] }
```

兼容文本形态：

```text
A,C
```

### 3. 判断题

判断题沿用点击式交互，但风格与单选 / 多选统一，而不是孤立实现。

设计要求：

- 提供“正确 / 错误”两个明确选项。
- 使用固定 token 保存，避免题库侧 `T/F` 与实验步骤侧 `TRUE/FALSE` 混用导致歧义。
- 学生侧统一使用实验步骤链路的 `TRUE / FALSE` 作为主 token。

结构化答案形态：

```json
{ "value": "TRUE" }
```

兼容文本形态：

```text
TRUE
```

### 4. 填空题

填空题必须改为**题干内联输入**，这是本次改造的核心体验目标之一。

设计要求：

- 按题干中的 `【填空1】`、`【填空2】` token 切分题干片段。
- 将 token 替换为内联下划线输入框。
- 输入框宽度允许根据内容与最小宽度折中，但不能因输入过长破坏整行布局。
- 支持多空位独立输入与回显。
- 题干中 token 顺序必须与答案顺序一一对应。

结构化答案形态：

```json
{
  "blanks": [
    { "index": 1, "answer": "foo" },
    { "index": 2, "answer": "bar" }
  ]
}
```

兼容文本形态：

```text
foo,bar
```

### 5. 简答题 / 文本题

主观文本题继续使用多行输入区，但不再与客观题共用提示和交互。

设计要求：

- 多行输入区支持合理的高度与字数反馈。
- 保留当前“保存当前题”节奏。
- 主观题提示语与客观题提示语分离。

结构化答案形态：

```json
{ "text": "学生输入内容" }
```

兼容文本形态：

```text
学生输入内容
```

### 6. 代码题

代码题继续使用代码编辑器，但要纳入统一答题壳体，而不是作为单独特例漂在页面中。

设计要求：

- 保留当前代码编辑器能力。
- 从详情接口返回代码题语言信息。
- 代码题保存结构与普通主观题分离，便于后续扩展。

结构化答案形态：

```json
{ "code": "print('hello')", "language": "PYTHON" }
```

兼容文本形态：

```text
print('hello')
```

## 四、前端状态模型设计

### 1. 状态目标

学生端不再使用 `Record<number, string>` 作为统一答案草稿模型，而改为按题型维护结构化作答状态。

### 2. 建议的统一前端答题模型

每道题维护如下结构：

```ts
type StudentAnswerDraft = {
  answerText: string
  answerPayloadJson: string
  typedAnswer:
    | { kind: 'single'; selectedKey: string | null }
    | { kind: 'multiple'; selectedKeys: string[] }
    | { kind: 'judge'; value: 'TRUE' | 'FALSE' | null }
    | { kind: 'fill'; blanks: Array<{ index: number; answer: string }> }
    | { kind: 'text'; text: string }
    | { kind: 'code'; code: string; language: string | null }
}
```

说明：

- `typedAnswer` 是页面真实交互态。
- `answerPayloadJson` 是保存接口使用的结构化 JSON 串。
- `answerText` 是兼容当前评分链的文本回退值。

### 3. 初始化规则

详情接口返回题项后，学生端按以下优先级初始化：

1. 若已有结构化学生答案，则优先 hydrate 为 `typedAnswer`。
2. 若只有旧 `answerText`，则按题型逆向解析为默认 `typedAnswer`。
3. 若两者都没有，则使用按题型的空态。

这样可以兼容历史数据与新数据并存阶段。

## 五、后端接口与契约设计

### 1. 学生实验详情接口

学生实验详情接口需要返回足够支撑答题 UI 的结构化元数据。

每个题项至少补充以下字段中的必要子集：

- `questionType`
- `content`
- `answerText`
- `answerPayloadJson`（若已有作答）
- `options`（单选 / 多选 / 判断）
- `blanks`（填空）
- `editorLanguage`（代码题）
- `questionSnapshotJson` 或其安全投影视图
- `answerConfigJson` 或其安全投影视图

设计原则：

- 学生端只拿“渲染答题必须知道”的字段，不透出教师端无关配置。
- 优先由后端完成投影和归一，避免前端直接消费原始混杂 JSON。

### 2. 保存单题接口

保存接口保持原地址不变，但语义升级。

请求体设计：

```json
{
  "answerText": "A,C",
  "answerPayloadJson": "{\"selectedKeys\":[\"A\",\"C\"]}"
}
```

保存规则：

1. `answerText` 继续保留，作为兼容字段。
2. `answerPayloadJson` 正式写入 `LabStepAnswer.answerJson`。
3. answer log 同步记录文本答案与结构化答案变更。

### 3. 提交实验接口

提交实验接口仍以 `summaryText` 为主，不在本次扩大到整卷统一 payload 提交。

理由：

- 当前页面已有“保存当前题”节奏。
- 每题答案已在切题与保存时落库。
- 本次重点是题型交互与单题结构化答案打通，而不是重做整卷提交协议。

## 六、后端持久化与兼容迁移

### 1. 持久化升级

`LabService.saveAnswer(...)` 需要从“只写 `answerText`”升级为：

- 写入 `answerText`
- 写入 `answerJson`
- 记录更新时间与日志

### 2. 详情回显兼容

学生详情接口回显时：

- 优先返回 `answerJson`
- 若历史记录为空，则仍返回 `answerText`
- 前端按优先级 hydrate

### 3. 评分兼容策略

本次不强行把整个实验评分链一次性切到结构化 JSON 优先，但需要明确迁移方向：

第一阶段：

- 评分主链继续兼容 `answerText`
- 新前端保存时同步生成标准化 `answerText`

第二阶段（未来可独立跟进）：

- 评分器优先读 `answerJson`
- 若不存在结构化答案，再回退读 `answerText`

这样可以避免本次改造把评分系统一并拉爆。

## 七、题型契约映射

### 1. 标准化学生结构化答案格式

本次学生端统一采用以下结构：

| 题型 | `answerPayloadJson` 结构 |
|---|---|
| 单选 | `{ "selectedKey": "A" }` |
| 多选 | `{ "selectedKeys": ["A", "C"] }` |
| 判断 | `{ "value": "TRUE" }` |
| 填空 | `{ "blanks": [{ "index": 1, "answer": "foo" }] }` |
| 简答/文本 | `{ "text": "..." }` |
| 代码 | `{ "code": "...", "language": "JAVA" }` |

### 2. 与实验步骤规则的映射

题目元数据来源于实验步骤规则，而学生答案来源于上述结构化学生作答 payload。

两者职责必须分离：

- `answerConfigJson`：题目规则 / 正确答案 / 评分规则
- `answerPayloadJson`：学生本次作答结果

不能再把两者都压成一个 `answerText` 字符串协议。

## 八、验证与验收标准

改造完成后必须同时满足以下验收标准：

1. 学生端实验答题页不再整页滚动。
2. 左侧题目导航区可独立滚动。
3. 右侧答题主体可独立滚动。
4. 单选题使用点击单选交互。
5. 多选题使用点击多选交互。
6. 判断题使用点击式二选一交互。
7. 填空题在题干文本中以内联输入框作答。
8. 简答题与代码题保留合理的文本/代码输入体验。
9. 保存后刷新页面可以正确回显结构化答案。
10. 后端持久化同时包含兼容文本与结构化 JSON。
11. 前端构建通过，后端测试或至少相关编译与联调通过。
12. Playwright 真实联调验证以下路径：进入实验、切题、保存、刷新回显、最终提交。

## 九、范围边界

本次改造明确包含：

- 学生端实验答题页布局重构
- 学生端分题型答题组件
- 学生端结构化答案状态模型
- 学生实验详情接口返回结构化元数据
- 学生保存接口真实持久化 `answerPayloadJson`

本次改造明确不包含：

- 教师端实验步骤编辑器再次重构
- 整个实验自动评分系统的全面重写
- 代码题在线运行、编译、执行沙箱
- 新增 mock 接口或假数据联调

---

## 十、实施建议

建议按以下顺序落地：

1. 先扩展后端详情 DTO 与保存逻辑，让结构化元数据与结构化答案链路成立。
2. 再改学生端类型与状态模型。
3. 然后重构学生答题页布局为双滚动工作台。
4. 最后逐题型替换交互组件并做联调验证。

这样可以把风险拆开，避免“UI 已写好但接口还不支持”或“接口已改完但前端仍按旧协议工作”的中间失配状态。

## 设计自检

- 本文没有保留 TBD / TODO / 待定项。
- 结构化题目元数据与结构化学生答案已明确区分。
- 迁移策略明确为“双写兼容”，没有要求一次性推倒评分系统。
- 实施范围聚焦在学生端实验答题闭环，没有扩散到教师端无关模块。
